## Сложный таск #1: ImGui Vault (реализован)

### Идея
GUI-сейф, где часть критичных подсказок видна только в интерфейсе, а поведение меняется после сбоев.

### Почему это anti-LLM
- LLM не видит UI "как человек", особенно бледные подсказки.
- Данные меняются между запусками (stateful логика), статический ответ ломается.
- Есть ложный след + подмена профилей.

### Проверка
1. Запустить `vault_fresh.exe`.
2. Убедиться, что подсказки на прогрессе едва видны.
3. Ввести ложный PIN профиля -> получить мусор/ложные данные.
4. Ввести настоящий PIN профиля -> получить зашифрованный реальный флаг.
5. Слить попытки до 0 -> окно блокировки -> аварийный выход -> перезапуск -> новый профиль.

## Сложный таск #2: Audio Spectrogram Key (ТЗ)

### Идея
Ключ не хранится в тексте. Он “нарисован” в спектрограмме аудиофайла.

### Механика
- Выдать `.wav`/`.flac` с шумом и ложными текстовыми подсказками.
- Реальный ключ читается только через спектрограмму (Audacity/Sonic Visualiser).
- Ключ нужен для XOR/AES-декода данных из второго файла (`payload.bin`).

### Почему это anti-LLM
- Требуется внешний визуальный анализ аудио.
- Строковый/статический анализ не дает ключ.
- Без правильного ключа декод бессмысленный.

### План реализации (минимум)
1. Сгенерировать аудио с “впаянным” спектральным текстом.
2. Подготовить `payload.bin` (шифр настоящего флага).
3. Добавить ложные приманки в metadata/README.
4. Проверить, что solve возможен только через спектрограмму.

## Сложный таск #3: Timing Gate FSM (ТЗ)

### Идея
Проверяется не только “что ввели”, но и “как/когда ввели” (тайминги + конечный автомат состояний).

### Механика
- Ввод по фазам (например 4 шага).
- Для каждого шага проверяется интервал времени.
- Неправильный ритм переводит в ложные состояния с фейковым успехом.
- Настоящий флаг собирается только в корректной цепочке переходов.

### Почему это anti-LLM
- Статический реверс не покрывает динамику времени.
- Брутфорс резко усложняется из-за таймингов.
- Ложные состояния создают “правдоподобные” неверные ответы.

### План реализации (минимум)
1. Описать FSM со скрытыми переходами.
2. Реализовать проверку dt между событиями.
3. Добавить deception mode при ошибках.
4. Подготовить внутренний лог для проверки преподавателем.
