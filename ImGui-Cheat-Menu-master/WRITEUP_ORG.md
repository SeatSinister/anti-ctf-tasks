# Eclipse Config (VOID CONFIG v2.1) — write-up для организаторов

## Что делает программа
Клиент — единственный «источник истины»: при нажатии **«ПРИМЕНИТЬ КОНФИГ»** считается условие, и открывается одно из окон:
- **[ УСПЕХ ]** (настоящее) с флагом из `kReal`
- **[ ОТКАЗ ]** (неудача) или **[ УСПЕХ ]** с декоем (ложная ветка) — в зависимости от значений в UI

Флаги — XOR’ом с ключом `0x5A`.
- Настоящий: `flqg{eclipse_config_vqlitqtet}`
- Декой/фейк: из `kFake`

## Как устроена проверка (анти-AI)
Проверка настоящего успеха основана на хэше FNV-1a по байтам кортежа состояния:
- `module_on` (checkbox)
- `fov` (little-endian: low byte, high byte)
- `mode_idx` (индекс пункта выпадающего списка)

В коде этот слой реализован как:
- FNV-1a в `vs::TmPacketHash(...)` (см. `void_seal.h`)
- константа-«запечатывание» разбита XOR’ом на части в `void_seal.h` + `void_seal_frag.h`
- настоящая проверка: `TmPacketHash(...) == sealed_ref`

Таким образом, простая подмена по «271» или «индексу режима» в лоб уже недостаточна: эталон не одна константа в `main.cpp`, а собирается из кусков.

## Подбор нужной тройки
Нужно найти `(module_on, fov, mode_idx)`, чтобы:
`TmPacketHash(module_on, fov, mode_idx) == SealedCfgRef()`.

Выполнение поиска в пределах UI-диапазонов даёт единственное попадание:
- `module_on = true`
- `fov = 271`
- `mode_idx = 4` (в UI это пункт **«Полный»**)

Важно: отдельный параметр интерфейса `noise_floor` не влияет на настоящую проверку.

## Ложная ветка (декой «успеха»)
В `main.cpp` есть вторая проверка, спроектированная под ложную подсказку в комментарии:
- `module_on = false`
- `fov = 180`
- `mode_idx = 0` (в UI это **«Выкл.»**)
- `noise_floor = 127`

Если выставить это и нажать **«ПРИМЕНИТЬ КОНФИГ»**, откроется окно с **канал-ловушкой** и текстом из `kFake`, который выглядит как «успех», но не является истинным флагом.

## Финальный флаг
При правильной тройке из раздела «Подбор нужной тройки» покажется:
`flqg{eclipse_config_vqlitqtet}`
